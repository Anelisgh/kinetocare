## 3.5 Infrastructură și containerizare: Docker, Kubernetes și persistența datelor

### 3.5.1 Containerizarea și paritatea mediilor de execuție (Docker)   
Pentru a garanta o tranziție predictibilă între mediul de dezvoltare și cel de producție, platforma KinetoCare utilizează Docker ca standard de containerizare. Această decizie arhitecturală elimină discrepanțele cauzate de sistemul de operare gazdă și rezolvă problema clasică a asimetriei dependențelor — un aspect critic într-un ecosistem cu șapte microservicii distincte, fiecare necesitând versiuni specifice ale mașinii virtuale Java și variabile de mediu proprii.   
**Optimizarea livrărilor prin construcție în etape multiple (*Multi-Stage Build*).** Pentru aplicația client, a fost implementată o strategie de asamblare în etape multiple. Aceasta separă net faza de compilare — care necesită un ecosistem complet Node.js pentru rezolvarea dependențelor și procesarea codului React — de faza de execuție. Imaginea finală de producție conține exclusiv fișierele statice rezultate (HTML, CSS, JavaScript minimizat) și este servită de un server web Nginx. Această abordare reduce drastic dimensiunea imaginii și minimizează suprafața de atac, eliminând din mediul de producție instrumentele de compilare și codul sursă brut.   
**Minimalismul imaginilor de *backend*.** Microserviciile Spring Boot sunt împachetate utilizând imagini de bază Alpine Linux cu un mediu de execuție Java (JRE) minimal. Alegerea unei distribuții Alpine în detrimentul unor imagini de sistem de operare complete reduce amprenta de memorie și vulnerabilitățile inerente sistemelor de operare generaliste.   
**Orchestrarea locală prin Docker Compose.** Pentru facilitarea ciclului de dezvoltare, ansamblul componentelor — microserviciile, furnizorul de identitate Keycloak, brokerul RabbitMQ și serverul MySQL — este orchestrat local prin Docker Compose. Acesta instanțiază o rețea virtuală dedicată (`kineto-network`) care asigură rezoluția DNS implicită, permițând serviciilor să comunice prin nume simbolice (ex. `mysql` sau `keycloak`), fără hardcodarea adreselor IP. Pentru a gestiona asincronismul pornirii serviciilor (ex. microserviciul de identitate depinde de un Keycloak complet inițializat), au fost implementate bucle de reîncercare programatice (*retry mechanisms*), o soluție rezilientă care evită eșecurile în cascadă la pornirea mediului de dezvoltare.   

### 3.5.2 Kubernetes: orchestrarea și scalabilitatea în mediul de producție   
Trecerea la mediul de producție implică cerințe superioare de auto-recuperare (*self-healing*), scalare orizontală și management distribuit al resurselor, aspecte delegate platformei Kubernetes.   
**Principiile arhitecturale Kubernetes aplicate:**   
- **Izolarea logică:** Toate componentele platformei sunt grupate într-un spațiu de nume (*Namespace*) dedicat (`kinetocare`), prevenind coliziunile de resurse cu alte aplicații găzduite în același cluster.   
- **Conformitatea cu metodologia 12-Factor App:** Configurațiile variabile (URL-uri interne, porturi) sunt extrase complet din codul sursă și administrate prin obiecte `ConfigMap`, în timp ce datele sensibile (parolele bazelor de date, secretele clienților OAuth) sunt stocate criptat în resurse de tip `Secret` cu acces restricționat (*Role-Based Access Control*). Această decuplare onorează principiul separării configurației de codul executabil.   
- **Persistența datelor (*Stateful Workloads*):** Deși containerele sunt efemere prin *design*, datele medicale și operaționale stocate în MySQL necesită persistență absolută. Aceasta este asigurată prin obiecte `PersistentVolumeClaim` (PVC), care decuplează ciclul de viață al stocării fizice de cel al containerelor (*pod*-urilor) — distrugerea unui container nu implică pierderea datelor.   
- **Mecanisme de auto-vindecare:** Microserviciile expun indicatori de sănătate prin Spring Boot Actuator, interogați constant de Kubernetes prin două tipuri de sonde:   
    - *Readiness Probes* validează capacitatea serviciului de a prelua trafic de rețea, asigurând că rutarea nu se face către un serviciu care încă își inițializează conexiunile la baza de date.   
    - *Liveness Probes* detectează stările de blocaj (*deadlocks*) și declanșează repornirea automată a containerelor nefuncționale.   
- **Global Edge Router (Ingress Controller):** Un controler Ingress gestionează traficul extern, acționând ca punct unic de intrare și de terminare TLS — un singur certificat de securitate acoperă întreaga platformă. Traficul este segmentat prin rutare bazată pe cale: cererile de autentificare sunt direcționate către Keycloak, traficul API către *Gateway* și cererile de interfață către serverul Nginx.   
   
*Notă privind degradarea grațioasă:* Proiectarea sistemului permite operarea unui subset redus de servicii (identitate, pacienți, terapeuți și API *Gateway*). Această configurație validează capacitatea de degradare grațioasă (*graceful degradation*): în lipsa modulului de programări, serviciile operaționale continuă să funcționeze, iar erorile de rețea (HTTP 502/503) sunt interceptate izolat la nivelul interfeței client prin *Error Boundaries*, prevenind prăbușirea întregii platforme.   

### 3.5.3 Arhitectura datelor: MySQL, izolare și denormalizare controlată   
Motorul relațional MySQL 8, cuplat cu motorul de stocare InnoDB, asigură fundamentul tranzacțional al platformei KinetoCare. Decizia a fost motivată de maturitatea ecosistemului, integrarea stabilă cu specificația JPA prin Hibernate și suportul robust pentru tranzacții ACID și blocare la nivel de rând (*row-level locking*), vitale pentru algoritmul de rezervare a programărilor.   
**Nivelul de izolare a tranzacțiilor**   
Sistemul utilizează nivelul de izolare `READ_COMMITTED`, asumat și documentat explicit la nivelul infrastructurii ca un compromis deliberat între consistență și performanță. Deși nivelul absolut `SERIALIZABLE` ar elimina garantat orice risc teoretic de condiții de cursă (*race conditions*) la rezervările concurente, acesta introduce penalizări de performanță inacceptabile și un risc crescut de blocaje reciproce (*deadlocks*) sub trafic intens. Riscul de suprapunere a sloturilor orare în condiții de concurență extremă sub `READ_COMMITTED` (în care două tranzacții simultane pot citi o stare liberă și insera în paralel) este eliminat prin utilizarea mecanismului de *blocare pesimistă* (*Pessimistic Locking*) prin clauza `SELECT ... FOR UPDATE` (adnotată prin `@Lock(LockModeType.PESSIMISTIC_WRITE)` în JPA) la interogarea de suprapuneri, blocând rândurile citite până la finalizarea tranzacției. Complementar, ca nivel secundar de siguranță împotriva înscrierilor concurente la aceeași secundă, baza de date implementează o constrângere de unicitate compusă (*UNIQUE KEY*) pe coloanele `(terapeut_keycloak_id, data, ora_inceput)`. Deși această constrângere la nivel de schemă previne exclusiv orele de începere identice (nereușind să blocheze suprapunerile parțiale de intervale, de exemplu un slot 10:00–11:00 suprapus cu 10:30–11:30), această limitare fizică este complet soluționată la nivelul logicii de *backend*. Microserviciul de programări integrează o verificare riguroasă a intersecției matematice a intervalelor orare (prin interogarea de suprapunere), securizată concurent prin blocarea pesimistă menționată anterior, ceea ce garantează eliminarea oricărei coliziuni sau duble rezervări în sistem.   
**Implementarea Database-per-Service și Cheile Externe Logice**   
În concordanță cu arhitectura de microservicii, platforma definește șapte scheme logice complet izolate. Deoarece principiile arhitecturii distribuite interzic accesul direct al unui serviciu la schema altuia, nu pot fi utilizate interogări SQL de tip `JOIN` între baze de date diferite. Arhitectura folosește în schimb identificatorul unic furnizat de Keycloak (`keycloakId`) ca pivot universal — o cheie străină logică, nu fizică — pentru a relaționa entitățile între servicii. Reîntregirea datelor se face prin apeluri API.   
**Denormalizarea controlată (Tiparul *Snapshot*)**   
Fragmentarea datelor necesită tipare de proiectare compensatorii. În platforma KinetoCare, la crearea unei programări, valorile financiare și temporale sunt denormalizate direct în entitatea programării conform tiparului *Snapshot*, a cărui justificare arhitecturală este detaliată în secțiunea 4.5.4.   

### 3.5.4 Sinergia componentelor: apărare stratificată în profunzime   
Analiza arhitecturii infrastructurii relevă aplicarea principiului *Defense in Depth* (apărare stratificată în profunzime) la trei niveluri fundamentale și complementare:   
1. **Izolarea la nivel de rețea:** Microserviciile nu sunt expuse direct internetului public. Traficul este canalizat și filtrat exclusiv prin axa Ingress Controller → API *Gateway*, reducând suprafața de atac la un singur punct de intrare strict controlat.   
2. **Izolarea la nivel de identitate:** Pe baza modelului *Zero-Trust*, perimetrul rețelei interne nu este considerat implicit sigur. Fiecare microserviciu aplică o revalidare criptografică independentă a jetoanelor JWT primite, fără a se baza pe garanțiile teoretice ale nivelului de rețea.   
3. **Izolarea la nivel de date:** Tiparul *Database-per-Service* limitează fatal raza de acțiune a unei eventuale breșe de securitate. Compromiterea ipotetică a serviciului de chat expune strict arhiva de conversații, lăsând intangibile dosarele medicale ale pacienților, stocate într-o schemă separată fizic și logic.   
   
Cele trei niveluri nu sunt independente, ci se consolidează reciproc: izolarea rețelei reduce probabilitatea unui atac, validarea descentralizată a identității elimină încrederea oarbă din interiorul perimetrului, iar izolarea datelor garantează că un defect local nu se transformă într-un compromis generalizat. Împreună, acestea formează o arhitectură de securitate coerentă și rezistentă la eșecuri sistemice.
