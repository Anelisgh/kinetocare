## Secțiunea 9: Arhitectura Frontend (Deep Dive)
Aplicația frontend KinetoCare este proiectată ca o aplicație SPA (Single Page Application) modernă și extrem de rezistentă. Funcționează ca interfață principală pentru pacienți, terapeuți și administratori, consumând microserviciile exclusiv prin intermediul API Gateway. Această secțiune detaliază deciziile arhitecturale, fluxurile de gestionare a stării și optimizările de interfață implementate pentru a susține activitatea clinică.
### 9.1 Arhitectura SPA & Structura de Directoare
Aplicația este construită folosind **React** în combinație cu **React Router v7** pentru navigare pe partea de client și rutare de date. Pentru a garanta scalabilitatea și separarea responsabilităților pe măsură ce aplicația crește, baza de cod urmează o structură strictă bazată pe funcționalități (features) și straturi (layers):
1. **/pages**: Conține componentele de rutare la nivel superior (e.g., `HomepageTerapeut.jsx`, `ChatPacient.jsx`). Acestea acționează ca containere care preiau date și orchestrează componentele copii.
2. **/layouts**: Implementează structuri persistente de interfață (e.g., `TerapeutLayout.jsx` care conține bara laterală și bara de navigare) prin intermediul mecanismului `<Outlet />` pus la dispoziție de React Router, prevenind re-randările inutile ale elementelor de navigare.
3. **/components**: Găzduiește fragmente refolosibile de interfață. Este subdivizat în funcție de domeniu (`/shared`, `/chat`, `/terapeut`, `/pacient`) pentru a menyin o coeziune ridicată.
4. **/services**: Stratul de comunicare cu API-ul. Fișiere precum `programariService.js` încapsulează toate apelurile HTTP Axios, izolând componentele UI de implementările REST directe.
5. **/context**: Conține furnizorii globali de stare, în special `AuthContext` pentru gestionarea identităților.
Această structură respectă Principiul Responsabilității Unice (Single Responsibility Principle), asigurând că componentele UI se ocupă doar de prezentare, în timp ce serviciile gestionează preluarea și normalizarea datelor.
### 9.2 State Management & Securitate (Stratul API)
Securitatea pe frontend este gestionată stateless. Pentru a preveni atacurile de tip Cross-Site Scripting (XSS), frontend-ul nu stochează niciodată tokenul de durată lungă `refresh_token` în `localStorage`. În schimb, acesta este administrat de API Gateway printr-un cookie de tip `HttpOnly`.
### 9.2.1 Starea de Identitate (`AuthContext.jsx`)
Componenta `AuthContext` oferă o stare globală a aplicației (`isAuthenticated` și `userInfo`).
La prima încărcare a aplicației SPA, aceasta intră într-o stare `isInitializing` și execută o reîmprospătare silențioasă (`authService.refreshToken()`). Browserul atașează automat cookie-ul `HttpOnly` la această cerere. Dacă backend-ul validează sesiunea, returnează un `access_token` de scurtă durată în memorie, permițându-i utilizatorului să rămână conectat la reîncărcarea browserului, fără intervenție manuală.
### 9.2.2 Interceptori Centralizați (`api.js`)
Toate cererile trimise în exterior sunt canalizate printr-o instanță personalizată de Axios, dotată cu interceptori puternici:
- **Request Interceptor (Interceptor de Cerere)**: Citește dinamic tokenul JWT aflat în memorie și injectează antetul `Authorization: Bearer <token>` în fiecare cerere trimisă spre exterior.
- **Response Interceptor (Bucla de Reîncercare la 401)**: Dacă un apel API eșuează cu eroarea `401 Unauthorized` (indicând că tokenul pe durată scurtă a expirat), interceptorul oprește respingerea cererii. Setează un flag `_retry = true` pentru a preveni buclele infinite, apelează endpoint-ul de silent refresh, actualizează tokenul local și re-execută transparent cererea HTTP originală. Pentru utilizatorul final, cererea de rețea pare doar să dureze cu câteva milisecunde mai mult.
- **Normalizarea Erorilor**: Backend-urile construite cu Spring Boot returnează structuri de tip `ProblemDetail` conforme cu standardul RFC 9457. Interceptorul analizează aceste payload-uri și construiește un obiect uniform JavaScript `AppError`. În mod specific, extrage mesajul `detail` și harta de validare la nivel de câmp `erori_campuri`, astfel încât formularele să poată asocia cu ușurință regulile de validare de pe backend direct cu câmpurile de intrare din pagină.
### 9.3 Rutare pe Bază de Roluri & Guards
Aplicația SPA impune controlul accesului bazat pe roluri (RBAC) în mod nativ în interiorul arborelui React Router folosind componente de tip Higher-Order (HOC).
### 9.3.1 `ProtectedRoute.jsx`
Această componentă învelește rutele sensibile. Interceptează încercările de navigare, verifică flag-ul `isAuthenticated` din `AuthContext` și confirmă dacă rolurile decodificate din JWT-ul utilizatorului se intersectează cu tabloul `allowedRoles`. Dacă utilizatorul nu este autentificat, declanșează `<Navigate to="/login" />`. Dacă este autentificat, dar nu este autorizat, îl redirecționează către `/unauthorized`.
### 9.3.2 Conformitatea Clinică (`ProfileGuard.jsx`)
Pacienții au permisiunea de a se înregistra doar cu email și parolă, dar nu pot rezerva programări sau folosi chat-ul fără a avea un profil medical complet (CNP, Data Nașterii, Date stil de viață).
`ProfileGuard.jsx` învelește întreaga structură a layout-ului `/pacient/*`. La montare, interoghează API Gateway. Dacă backend-ul semnalează `profileIncomplete === true` (sau dacă se detectează local că `cnp` este null), guard-ul blochează accesul la dashboard și forțează o redirecționare către asistentul de completare a profilului (`/pacient/complete-profile`).
### 9.4 Fluxuri Clinice & de Planificare (Interfața Terapeutului)
Interfața de planificare este cea mai complexă componentă a frontend-ului, necesitând validarea suprapunerilor în timp real și filtrare inteligentă.
### 9.4.1 Integrarea `TerapeutCalendar.jsx`
Programul terapeutului este vizualizat cu ajutorul `@fullcalendar/react`.
1. **Preluare Dinamică pe Bază de ISO**: Calendarul nu preia toate programările istorice. În schimb, funcția `events` din FullCalendar trimite limitele curente ale ecranului vizualizat (`startStr` și `endStr` în format ISO-8601) către metoda `programariService.getCalendarAppointments`.
2. **Filtrarea Inteligentă a Evenimentelor**: Frontend-ul aplică o euristică strictă de filtrare. Programările marcate ca `ANULATA` sunt eliminate din listă *cu excepția* cazului în care `motivAnulare === 'NEPREZENTARE'`. Acest lucru asigură că ecranul terapeutului nu este supraîncărcat cu ședințe anulate în mod legitim, păstrând în același timp istoricul vizual al pacienților care nu s-au prezentat.
3. **Randare Personalizată**: Hook-ul `renderEventContent` aplică clase CSS dinamice pe baza proprietăților din `extendedProps`. De exemplu, dacă `primaIntalnire` este adevărată, adaugă clasa `status-programata prima-intalnire`, colorând blocul în galben. Acest lucru îl alertează instantaneu pe terapeut că trebuie să aplice protocolul de „Evaluare Inițială”.
### 9.5 Chatul în Timp Real & Integrarea STOMP
Modulul de chat oferă o interfață cu panouri împărțite (split-pane) pentru comunicarea securizată, în timp real, între pacienții și terapeuții autorizați.
### 9.5.1 Ciclul de Viață WebSocket (`ChatTerapeut.jsx`)
1. **Agregare BFF**: La montare, componenta efectuează un singur apel HTTP optimizat (`chatService.getConversatiiAgregate`) către API Gateway. Acesta returnează o listă ce conține identitatea partenerului, cel mai recent fragment de mesaj și statusul privind arhivarea, eliminând problema interogărilor N+1.
2. **SockJS & STOMP**: Componenta inițializează un client `@stomp/stompjs` Client. Deoarece conexiunile standard WebSocket nu permit transmiterea de anteturi HTTP personalizate în mod nativ în API-urile browserelor, conexiunea este stabilită printr-un wrapper de tip fallback `SockJS`, permițând transmiterea securizată a tokenului JWT în `connectHeaders`.
3. **Reactivitate**: Atunci când un cadru de mesaj este primit pe canalul STOMP la care utilizatorul este abonat, starea React este actualizată, adăugând instantaneu mesajul în DOM și derulând automat vizualizarea până la capătul de jos. Erorile (cum ar fi încercarea de a trimite un mesaj unui pacient dezactivat) sunt prinse prin intermediul callback-ului `onStompError` și sunt afișate sub formă de alerte în interfață.
### 9.6 Panoul Administrative de Business Intelligence
Panoul de administrare (`AdminStatistici.jsx`) funcționează ca un motor puternic de agregare a datelor.
1. **Ingestie în Paralel**: Când intervalul de date se modifică, componenta declanșează un bloc `Promise.all`, preluând concurent 6 seturi de date distincte (Programări Lunare, Venituri per Locație, Rate de Anulare, Pacienți Noi, Ture Terapeuți).
2. **Calcule Memoizate**: Indicatorii Cheie de Performanță (KPIs) de nivel superior, cum ar fi Venitul Total sau Rata Medie de Anulare, sunt costisitori din punct de vedere al resurselor de calcul. Aceștia sunt înveliți în hook-ul `useMemo` din React. Când un administrator selectează o anumită locație din meniul derulant, matricele de date brute sunt refiltrate, dar `useMemo` garantează că indicatorii se recalculează instantaneu, fără a declanșa o re-randare completă a nodurilor grafice.
3. **Integrarea Recharts**: Datele pivotate sunt transmise componentelor din `recharts`. Aplicația SPA randează grafice de tip bară stivuite (`BarChart`) pentru a afișa programările per locație în timp și grafice de arie (`AreaChart`) pentru a vizualiza viteza de achiziție a pacienților, utilizând gradiente pe bază de imagini SVG pentru un aspect premium.
### 9.7 Reziliența UI & Performanța
Având în vedere natura clinică a aplicației, erorile silențioase (silent failures) sunt inacceptabile.
### 9.7.1 Componenta `ErrorBoundary.jsx`
În trecut în React, o excepție netratată în faza de randare (e.g., încercarea de a mapa o listă nulă returnată de un API) demonta întreaga aplicație, rezultând un ecran alb de eroare (White Screen of Death).
Pentru a preveni acest lucru, arborii de componente critici sunt înveliți într-o componentă clasă `ErrorBoundary`. Aceasta implementează metodele de ciclu de viață `componentDidCatch` și `getDerivedStateFromError`. Dacă o componentă copil se prăbușește, componenta de graniță prinde excepția, înregistrează stiva de apeluri în consolă și afișează o interfață elegantă de fallback („A apărut o eroare neașteptată”) care conține un buton de acțiune „Reîncarcă pagina”, asigurându-se că utilizatorul nu rămâne niciodată în fața unui ecran gol și blocat.
### 9.7.2 Polling Inteligent al Resurselor (Page Visibility API)
Componenta `NotificationBell` implementează interogarea HTTP periodică în fundal (la fiecare 30 de secunde) pentru a prelua numărul de notificări necitite. Cu toate acestea, interogarea naivă poate consuma inutil bateria dispozitivului și resursele de pe backend dacă utilizatorul lasă tab-ul deschis în fundal timp de mai multe ore.
Pentru a optimiza acest lucru, componenta se integrează cu API-ul nativ al browserului **Page Visibility API** (`document.visibilityState`).
- Atunci când tab-ul își pierde focalizarea (`hidden`), un hook `useEffect` din React interceptează evenimentul și oprește intervalul de polling.
- Atunci când utilizatorul revine la tab-ul KinetoCare (`visible`), intervalul este restabilit automat și este declanșat un apel instantaneu `fetchCount()` pentru a se asigura că interfața reflectă imediat orice eveniment petrecut în lipsa utilizatorului.
Acest lucru oferă o experiență în timp real, conservând în același timp lățimea de bandă a rețelei și capacitatea de procesare a API Gateway.
### 9.8 Fluxurile Pacienților & Interfețele Clinice
Pe lângă instrumentele destinate terapeuților și administratorilor, aplicația SPA oferă un portal dedicat, extrem de reactiv, pentru ca pacienții să își poată gestiona procesul de recuperare, alături de interfețele clinice folosite de terapeuți pentru a documenta această recuperare.
### 9.8.1 Dashboard-ul Pacientului (`HomepagePacient.jsx`)
Pagina principală a pacientului este ecranul principal de destinație, oferind un rezumat complet al planului curent de tratament.
- **Urmărirea Progresului Tratamentului**: Interfața UI extrage valorile `situatie.sedinteEfectuate` și `situatie.sedinteRecomandate` din răspunsul API-ului. Calculează dinamic un procent și randează o bară de progres bazată pe CSS. Dacă `sedinteEfectuate >= sedinteRecomandate`, afișează automat un mesaj de felicitare și recomandarea de a programa o reevaluare.
- **Zona de Acțiune Ternară**: Zona principală de acțiune a panoului funcționează ca un automat de stări finite pe baza contextului pacientului:
1. *Niciun Terapeut Selectat*: Randează componenta `<TerapeutSection />`, invitând pacientul să își aleagă un terapeut (ceea ce declanșează inițializarea relației pe backend).
2. *Există o Programare Activă*: Randează componenta `<ActiveAppointmentCard />`, afișând detaliile ședinței viitoare și expunând o opțiune de anulare în timp real (`handleCancel`).
3. *Modul de Rezervare*: Dacă pacientul are un terapeut selectat, dar nu are nicio programare viitoare, randează componenta `<BookingWidget />`, inițiind fluxul de planificare.
### 9.8.2 Fluxul de Programare al Pacientului (`ProgramariPacient.jsx`)
Ecranul `ProgramariPacient` extinde zona de acțiune a dashboard-ului prin atașarea unui istoric complet al programărilor (`<HistoryAppointmentCard />`).
Atunci când un pacient interacționează cu componenta `<BookingWidget />`, acesta alege o dată și o oră. Frontend-ul gestionează disponibilitatea în timp real prin interogarea `programariService`. Dacă un pacient dorește să anuleze o ședință, funcția `handleCancel` solicită o confirmare din partea browserului (`window.confirm`) și execută un apel API non-blocant. După finalizarea cu succes, declanșează `refreshDashboard()` printr-un hook `useCallback` din React, readucând instantaneu interfața la Modul de Rezervare, fără a fi nevoie de reîncărcarea paginii.
### 9.8.3 Fișa Pacientului (`FisaPacient.jsx`)
Această componentă complexă destinată terapeuților reprezintă sursa definitivă de adevăr pentru istoricul medical al unui pacient.
- **Ingestie Concurentă**: La montare, execută un bloc `Promise.all` pentru a prelua simultan fișa clinică a pacientului (`getFisaPacient`) și tendința feedback-ului său subiectiv (`getJurnalTrend`).
- **Segregarea Datelor**: Interfața UI se bazează intens pe randarea condiționată pentru a implementa o interfață cu Tab-uri, izolând secțiunile `Evaluări`, `Evoluții` (Note de Progres), `Programări` (Istoric Ședințe) și `Jurnale` (Feedback Subiectiv). Acest lucru previne supraîncărcarea cognitivă a terapeutului.
- **Vizualizarea Tendințelor Subiective**: Componenta `JurnalEvolutieChart` consumă matricea de date `trendData` și randează un grafic liniar `LineChart` din biblioteca `Recharts`. Mapează valorile `nivelDurere` (Durere), `dificultateExercitii` (Dificultate) și `nivelOboseala` (Oboseală) pe o axă Y cu valori de la 1 la 10, permițându-i terapeutului să vizualizeze instantaneu dacă starea pacientului se îmbunătățește sau se deteriorează pe parcursul ultimelor 10 ședințe.
### 9.8.4 Formularele Clinice (Evaluare & Jurnal)
Aplicația SPA se bazează puternic pe modelul de **Componente Controlate (Controlled Components)** din React pentru a impune integritatea datelor înainte ca payload-urile să ajungă la API Gateway.
- **Evaluări (`EvaluariTab`)**: Atunci când un terapeut modifică o evaluare, starea locală (`editData`) impune legături (bindings) stricte. Listele derulante pentru `serviciuRecomandatId` sunt populate dinamic din `evaluariService.getAllServicii()`. Câmpul de intrare pentru `sedinteRecomandate` impune constrângerile HTML5 `min="1"` și `max="50"`, garantând că payload-urile malformate nu pot fi trimise spre server.
- **Jurnalul Pacientului (`JurnalPacient.jsx`)**: Interfața UI apelează `getSedinteNecompletate` și forțează pacientul să completeze feedback-ul pentru ședințele ratate. Formularul înlocuiește câmpurile numerice clasice cu glisoare de tipul `<input type="range" />` limitate la scara 1-10. Acest lucru reduce semnificativ efortul utilizatorilor de pe dispozitive mobile, capturând cei 3 indicatori principali (Durere, Dificultate, Oboseală) alături de un câmp opțional de text liber `comentarii` înainte de trimitere.
### 9.9 Mecanismul de Livrare Frontend & Containerizarea
În timp ce mediul de dezvoltare folosește funcționalitatea de înlocuire rapidă a modulelor (HMR) oferită de Vite, mecanismul de livrare pentru producție este puternic optimizat printr-un **proces multi-stage de build în Docker**:
1. **Etapa de Build**: Un container Node.js instalează dependențele și execută procesul de build al Vite. Acest lucru compilează întreaga aplicație React în pachete statice de fișiere HTML, JavaScript și CSS.
2. **Etapa de Producție**: Elementele statice compilate sunt copiate într-un container ușor de **Nginx**. Mediul de execuție Node.js de mari dimensiuni și instrumentele de dezvoltare sunt eliminate.
Această strategie multi-stage produce o imagine de producție extrem de mică, rapidă și sigură. Nginx funcționează ca un server eficient de fișiere statice, în timp ce Kubernetes Ingress gestionează terminarea TLS și rutarea dinamică înainte ca traficul să ajungă la containerul Nginx.
